Define and enforce development standards for AI-assisted programming

Why Standards Matter for AI Development

When working with AI assistants like Claude, setting clear, explicit standards upfront is crucial for:

  • Consistency: Ensuring all code follows the same patterns and conventions
  • Quality: Maintaining high standards across all AI-generated code
  • Maintainability: Creating code that's easy to understand and modify
  • Team Alignment: Everyone (human and AI) follows the same rules
  • Automated Enforcement: Standards can be validated automatically
üéØ Key Principle: Front-load your standards definition. It's much easier to establish standards before AI generates code than to refactor afterwards.

    validation: [validate, verify, check, ensure, assert, confirm]

    notification: [send, notify, email, alert, publish, broadcast]

Front-Loading Standards for AI

1. Pre-Development Standards Definition

üìù Documentation First

Write comprehensive specs before asking AI to code. Include examples of desired patterns.

üèóÔ∏è Architecture Patterns

Define specific design patterns, file structures, and naming conventions upfront.

üß™ Test-Driven Standards

Write tests first that encode your quality standards, then have AI implement.

üîß Tool Configuration

Set up linters, formatters, and quality gates before generating code.

üí° Pro Tip: Create a CLAUDE.md file in your repository with your standards and preferences. AI assistants can reference this automatically.
üìù Real Example: CLAUDE.md for AI Instructions
# CLAUDE.md - AI Assistant Instructions

## Code Standards
- All Python code must achieve Radon Grade A complexity
- Use type hints for all function parameters and returns
- Follow PEP 8 and use Black formatting
- No print statements - use logging instead

## Architecture Standards
- Follow SOLID principles (enforced by .design-lint.yml)
- Maximum 7 methods per class
- Maximum 3 responsibility groups per class
- Dependency injection required for all services

## Testing Standards
- Minimum 80% code coverage
- All new features must have unit tests
- Integration tests for API endpoints
- Use pytest for Python, Jest for JavaScript

## Documentation Standards
- All files must have header comments with purpose and scope
- Functions need docstrings with parameter descriptions
- Complex logic requires inline comments
- Update README when adding new features

2. Effective Prompting Strategies

  • Be Specific: "Follow SOLID principles" vs "Use dependency injection with constructor parameters"
  • Provide Examples: Show the exact patterns you want AI to emulate
  • Reference Standards: Point to your linting configs and style guides
  • Set Constraints: "Functions must be under 20 lines" or "Use TypeScript strict mode"

Security Check Example

Here's how to integrate Claude directly into your GitHub Actions for security analysis:

üîí Real Example: Claude Security Review Action
name: AI Security Review

on:
  pull_request:
    branches: [main]

jobs:
  claude-security-check:
    name: Claude Security Analysis
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Get changed files
        id: changed-files
        run: |
          git diff --name-only ${{ github.event.pull_request.base.sha }} ${{ github.sha }} > changed_files.txt
          
      - name: Run Claude Security Analysis
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          # Create security analysis script
          cat > security_check.py << 'EOF'
          import os
          import anthropic
          
          client = anthropic.Client(api_key=os.environ['ANTHROPIC_API_KEY'])
          
          # Read changed files
          with open('changed_files.txt', 'r') as f:
              changed_files = f.read().splitlines()
          
          security_issues = []
          
          for file_path in changed_files:
              if file_path.endswith(('.py', '.js', '.ts', '.jsx', '.tsx')):
                  try:
                      with open(file_path, 'r') as f:
                          file_content = f.read()
                      
                      response = client.messages.create(
                          model="claude-3-sonnet-20240229",
                          max_tokens=1000,
                          messages=[{
                              "role": "user", 
                              "content": f"""
                              Analyze this code for security vulnerabilities:
                              
                              File: {file_path}
                              ```
                              {file_content}
                              ```
                              
                              Look for:
                              - SQL injection vulnerabilities
                              - XSS vulnerabilities  
                              - Authentication/authorization issues
                              - Input validation problems
                              - Secrets or credentials in code
                              - Unsafe deserialization
                              
                              Return ONLY a JSON object with:
                              {{"issues": [list of security issues], "severity": "high|medium|low|none"}}
                              """
                          }]
                      )
                      
                      # Process Claude's response (would need proper JSON parsing)
                      print(f"Security analysis for {file_path}: {response.content}")
                      
                  except Exception as e:
                      print(f"Error analyzing {file_path}: {e}")
          EOF
          
          python security_check.py > security_report.txt
          
      - name: Comment on PR
        if: always()
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('security_report.txt', 'utf8');
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## üîí Claude Security Analysis\n\n\`\`\`\n${report}\n\`\`\``
            });

Real-Life Scenarios

Scenario 1: Onboarding New AI Assistant

Situation: Your team starts using a new AI assistant for development.

Solution with Standards:
  1. Share your CLAUDE.md configuration file
  2. Run the AI through your linting pipeline on first commit
  3. Review and adjust based on violations
  4. Document any new patterns that emerge

Scenario 2: Legacy Code Refactoring

Situation: You need AI to help refactor a large legacy codebase.

Solution with Standards:
  1. Create target architecture documentation
  2. Set up design principle linters (like our SRP analyzer)
  3. Have AI refactor in small chunks, validating each against standards
  4. Use trend analysis to ensure metrics improve over time

Scenario 3: Multi-Language Project

Situation: Your project uses Python backend, TypeScript frontend.

Solution with Standards: 
# Our actual multi-language setup
backend/
  - Black, isort, ruff, mypy, pylint, bandit
  - Radon Grade A complexity enforcement
  - Design principle analyzers

frontend/
  - ESLint, Prettier, TypeScript strict mode
  - Component pattern enforcement
  - Import organization standards

# Unified reporting in CI/CD
quality-gate:
  - Combines results from all language checks
  - Blocks merge if any standards violated
  - Generates comprehensive reports

Scenario 4: Security-First Development

Situation: Building a financial application where security is paramount.

Solution with Standards:
  • Pre-commit Security: Bandit catches issues before code review
  • AI Security Review: Claude analyzes every PR for vulnerabilities
  • Dependency Scanning: Safety and npm audit in CI pipeline
  • Custom Rules: Project-specific security patterns in linting config
